# MVC

## Pattern Architetturale

Serve a dare una buona separaizone delle funzionalita'
devo fare in modo che se una parte di codice non funziona non devo andare a ricontrollare troppo codice.

Devo frammentare il codice

### View, contorller e Model

- Il controller gestisce la logica di controllo: ad esempio spostare una carta
- La view si interfaccia col giocatore e gli moostra il gioco
- Il model e' la rappresentazione dello stato del sistema.
  In questo caso lo stato e' la board di gioco con le carte, l'elenco dei giocatori, le carte del mazzo, rappresenta la struttura del gioco da tavolo.

### tipica azione

- Gli utenti interagiscono con la loro view che ci fa vedere le cose
- La view inoltra la mia volonta' al controller.

il contorller ha tutte le possibili azioni che si possono fare nel gioco

- il contorller modifica il model.

In questo momento non e' cambiato niente:
Devo modificare la view per **tutti** gli utenti

- aggiorno la view

Nel model mettiamo le modifiche dirette dello stato, non andiamo ad implemetare la logica di gioco, che invece e' gestita dal contorller.

Il model ha i metodi e gli oggetti per modificare la struttura dati.

Certi pezzi di logica possono essere nel model, ad esempio negli scacchi se voglio spostare una pedina fuori dalla board e' il model che lo impedisce.

Non posso chiedere all'utente che cosa deve fare, infatti l'utente deve poter fare piu' di una cosa alla volta.
Client e server devono essere pronti a gestire eventi.

Il controller e' sia sul server che sul client.

### Porgrammazione ad eventi

- observer
- observable
- eventi

Quando faccio qualcosa e clicco un pulsante non aspetto l'effetto del pulsante.

#### differenti approcci

- no controller sul client: pagine web in cui non faccio nulla se non cliccare link, il controller e' solo sul server
- controller sul client mi permette di avere pulsanti e cose da fare

La view sta sul Client, model e controller sul server.

#### virtual server e virtual view

componenete aggiuntivo che ci permette di chiamare come se stessimo chiamando le funzioni dal server.
vogliamo spostore una carta:

1. facciamo la chiamata dal client con i parametri corretti passandola al virtual server
2. il virtual server chiama la virtual view( che sara' un thread)
3. mando il messaggio al controller che ha la funzione vera.
4. il controller modifica il model.
5. il model viene modificato e aggiorna la virtual view

- c'e' una virtual view per ogni client

## MVC e Rete

Server e client devono gestire piu' cose in simultanea.

Mai bloccare il thread o l'interfaccia. Le cose devono essere modificate da qualcosa di esterno in modo che se vogliamo possiamo sempre stampare il dato.
